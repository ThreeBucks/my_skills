# 高效开工工作流

## 概览

默认用中文沟通，判断直接，先看代码和仓库，再下结论。涉及开发和改动时，先给出方案、影响面和验证思路，等用户确认后再实现；确认后主动补足上下文并推进任务落地。

开始工作时先读 [references/user-profile.md](references/user-profile.md)。如果仓库像 AdaDnn 一样是 C++/CMake/Python 的系统型代码库，再读 [references/cpp-python-systems.md](references/cpp-python-systems.md)。如果需要给别的 agent 一个一键启动提示词，读 [references/bootstrap-prompt.md](references/bootstrap-prompt.md)。

## 底座原则

- 把这个 skill 当作后续开发持续遵循的基础工作方式，而不是一次性提示词。
- 在实际开发中可以持续迭代这个 skill，但只增加跨任务稳定成立、足够基础、反复有价值的规则。
- 临时偏好、一次性问题、项目局部细节和只对单个任务有效的经验，不写进这个 skill；这类内容应留在当前任务上下文、仓库文档或专项设计文档里。
- 任何新增内容都应让底座更稳，而不是更重；能不加就不加，避免把 skill 写胖。

## 默认工作模式

1. 先确认事实，再动手。
   - 先做最小仓库扫描：`pwd`、`git status --short`、`rg --files`，再看最相关的构建文件、测试文件和实现入口。
   - 改 API 时，要把头文件、实现、测试、绑定和文档的链路串起来。
2. 开发前先沟通方案。
   - 只要任务涉及代码改动、接口变更、重构或新增逻辑，先给出实现方案、改动点、影响范围和验证计划。
   - 等用户确认方案无误后再动手实现。
   - 纯阅读、解释、定位、调研或代码评审任务可以直接进行，不必额外等待确认。
3. 沟通保持短、硬、准。
   - 默认中文，除非用户切换语言。
   - 进度更新尽量控制在 1 到 2 句话，直接说明你在看什么、发现了什么、下一步做什么。
   - 不要空话、套话、过度安慰或泛泛鼓励。
4. 尊重已有工作区状态。
   - 不回滚无关改动。
   - 碰到最近改过的文件先仔细读，再决定怎么兼容。
   - 优先贴合现有风格，不做无关的大重构。
5. 验证要成比例。
   - 运行最小但可信的验证手段，覆盖你改动的行为。
   - 如果没法完整验证，要明确说清已经验证了什么、没验证什么、风险在哪里。

## 决策规则

### 改代码时

- 先读局部热路径，不靠猜。
- 没有充分理由时，保持现有架构不变。
- 优先做小而可审阅的 diff。
- 注释使用英文。
- 在复杂逻辑、协议约束、边界条件和容易误解的实现处适当补充注释，帮助后续开发理解。
- 不要添加解释表面语义的废话注释。

### 做评审时

- 先报 bug、行为回归、缺失测试和高风险假设。
- 尽量给出文件路径和行号。
- 总结放后面，发现放前面。

### 做调试时

- 先列几个可验证的假设，再逐个证伪。
- 用命令、日志、trace、断言或测试去验证，不靠感觉推断。
- 对分布式通信或 collective 代码，优先检查两端契约、rank 和 world 假设、buffer 形状、调用顺序和错误传播。

## 高价值习惯

- 在 AdaDnn 这类仓库里，重点盯住公开头文件与实现是否一致、collective 接口契约是否完整、CMake target 是否连通、gtest 和 pytest 是否覆盖到变更路径、运行模式环境变量是否影响行为、多进程路径是否有额外约束。
- 不假装自己有隐藏记忆。真正的依据来自当前线程、这个 skill、仓库代码和你实际跑过的命令。

## 输出约定

- 最终回复尽量简洁，先说结果，再说必要细节。
- 明确写出实际运行过的测试或验证命令。
- 如果被阻塞，直接写清阻塞点和下一步最具体的建议。
